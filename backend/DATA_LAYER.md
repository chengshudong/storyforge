## Repository Pattern

All repositories extend `BaseRepository[T]`:

```python
class BaseRepository[T]:
    def __init__(self, session: AsyncSession, model: type[T]): ...
    async def create(self, entity: T) -> T       # add + flush
    async def get(self, id: UUID) -> T | None     # pk lookup
    async def list(self, offset, limit) -> list[T] # paginated
    async def update(self, entity: T) -> T        # merge + flush
    async def delete(self, entity: T) -> None     # remove + flush
    async def count(self) -> int                  # scalar count
```

Rules:
- Repositories only call `flush()`, never `commit()`
- Transaction boundaries managed by `get_db()` dependency
- No raw SQL in services — all queries in repository methods

## Transaction Flow

```
Request
  → get_db() yields AsyncSession
    → Handler calls Service
      → Service calls Repository
        → Repository calls session.flush()
  → get_db() commits on success / rollbacks on exception
Response
```

## Migration

```bash
cd backend
alembic upgrade head     # apply all migrations
alembic downgrade -1     # rollback one version
alembic current          # show current version
alembic history          # show migration chain
```

Single baseline migration `001_initial_schema.py` creates all tables and enum types. Hand-written (not autogenerated) for precise control over native PostgreSQL enums.

## Key Design Decisions

1. **UUID v4 primary keys** — distributed-friendly, no sequence contention
2. **Native PostgreSQL ENUMs** — type safety at the database level
3. **server_default on timestamps** — `now()` handled by DB, not application
4. **Lazy engine initialization** — no DB connection at import time
5. **Explicit enum type names** — `Enum(ProjectStatus, name="project_status")` to align model ↔ migration ↔ PostgreSQL
